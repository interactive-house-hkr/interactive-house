1. Introduction

This API contract defines the communication between the Interactive House mobile/web client and the server. The purpose is to ensure that both teams implement compatible functionality and use consistent request and response formats.

This document specifies:

  • Endpoints
  • Request/response formats
  • Validation and error responses
  • Device model
  • Command structure
  • Authentication behavior
  • Additional technical notes

2. Base URL and versioning

All implemented endpoints are versioned under the following prefix:

/api/v1

This means that earlier conceptual routes such as /devices are implemented as /api/v1/devices.

3. Authentication

3.1 Login

Endpoint

POST /api/v1/auth/login

Request Body

{
  "username": "user1",
  "password": "password123"
}

Validation Rules (Confirmed)

  • username is required
  • password is required
  • Missing required fields return 400 Bad Request
  • Authentication failures return 401 Unauthorized
  • The API accepts username-based login in this contract

Success Response

{
  "status": "success",
  "token": "jwt_token_here",
  "userId": "123"
}

Error Responses (Confirmed)

{
  "status": "error",
  "error": "missing_fields",
  "message": "username and password are required"
}
{
  "status": "error",
  "error": "invalid_credentials",
  "message": "invalid username or password"
}
{
  "status": "error",
  "error": "server_error",
  "message": "unexpected server error"
}

3.2 Registration

Endpoint

POST /api/v1/auth/register

Request Body

{
  "username": "user1",
  "email": "user1@example.com",
  "password": "password123"
}

Required Fields (Confirmed)

  • username
  • email
  • password

Validation Rules (Confirmed)

  • All required fields must be present
  • email must have valid email format
  • username must be unique
  • email must be unique
  • Invalid input returns 400 Bad Request
  • Duplicate user data returns 409 Conflict

Success Response

{
  "status": "success",
  "userId": "123",
  "token": "jwt_token_here"
}

Error Responses (Confirmed)

{
  "status": "error",
  "error": "missing_fields",
  "message": "username, email and password are required"
}
{
  "status": "error",
  "error": "invalid_email",
  "message": "email format is invalid"
}
{
  "status": "error",
  "error": "username_exists",
  "message": "username already exists"
}
{
  "status": "error",
  "error": "email_exists",
  "message": "email already exists"
}
{
  "status": "error",
  "error": "server_error",
  "message": "unexpected server error"
}

4. Devices

4.1 List Devices

Endpoint

GET /api/v1/devices

Example Response

[
  {
    "device_uuid": "light-1",
    "name": "Living Room Lamp",
    "room": "living_room",
    "type": "light",
    "capabilities": {
      "power": {
        "type": "boolean",
        "writable": true
      },
      "brightness": {
        "type": "integer",
        "min": 0,
        "max": 100,
        "writable": true
      }
    },
    "state": {
      "power": true,
      "brightness": 80
    }
  }
]

Supported device types (Confirmed)

  • controller
  • light
  • fan
  • Other device types may be forwarded by the bridge without hardcoding, but these are the explicitly documented types from the device-side schema.

4.3 Device model

The client-facing API uses a single state object.

FieldDescription
device_uuidUnique identifier of the device
nameDisplay name shown in the client
roomOptional room/category location
typeDevice type, for example controller, light or fan
capabilitiesObject describing which controls can be rendered by the client
stateCurrent state of the device

4.4 Capabilities format

Capabilities supported according to the device-side schema

CapabilityTypeWritableNotes
powerbooleanYesUsed by light devices
brightnessintegerYesRange 0-100. Device-side schema models this as a range capability.
speedintegerYesUsed in command examples for fan devices
doorbelleventNoExample event capability from the device-side schema

Client-facing format example

{
  "capabilities": {
    "power": {
      "type": "boolean",
      "writable": true
    },
    "brightness": {
      "type": "integer",
      "min": 0,
      "max": 100,
      "writable": true
    }
  }
}

4.5 State format

Example

{
  "state": {
    "power": true,
    "brightness": 80
  }
}

Confirmed behavior

  • The API returns one state object per device
  • State keys depend on the device type and supported capabilities
  • If a device does not support a field, that field is omitted rather than returned as null

5. Device status updates

Confirmed mechanism

Polling

For the initial version of the client contract, the server will not implement WebSocket, Server-Sent Events (SSE) or MQTT bridge integration for real-time updates. Instead, clients should poll the device endpoints to fetch the latest state.

Clients can still fetch updates using:

  • GET /api/v1/devices
  • GET /api/v1/devices/{device_uuid}

Recommended interval

5 seconds (only if polling is used)
WebSocket, SSE and MQTT bridge integration remain possible future options, but they are not part of the confirmed current client contract.

6. Commands

6.1 Send Command to Device

Endpoint

POST /api/v1/devices/{device_uuid}/commands

Request Body

{
  "state": {
    "power": true
  }
}

Required fields (Confirmed)

  • device_uuid in the route path
  • state in the JSON body
  • The body must contain at least one state field to update

Example Success Response

{
  "status": "success",
  "state": {
    "power": true
  }
}

Error Responses (Confirmed)

{
  "status": "error",
  "error": "missing_state",
  "message": "state is required"
}
{
  "status": "error",
  "error": "not_found",
  "message": "device not found"
}
{
  "status": "error",
  "error": "unsupported_capability",
  "message": "device does not support one or more requested state fields"
}
{
  "status": "error",
  "error": "invalid_state",
  "message": "one or more state values are invalid"
}
Earlier draft examples showed the command body as only { "power": true }. This updated version uses a wrapped state object to make the command format explicit and consistent.

9. Technical constraints

Timeout limits (Confirmed)

  • Normal API requests should complete quickly under standard local-network conditions
  • Client applications should use a practical request timeout around 5-10 seconds

Rate limits (Confirmed)

  • No explicit rate limiting is defined in this contract version
  • Clients should still avoid excessive polling beyond the recommended interval

Required headers

Authorization: Bearer <token> is not required for login or registration api

Authorization: Bearer <token>
Content-Type: application/json
