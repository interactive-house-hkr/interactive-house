REST Device - Server API Contract

This document describes how a REST-based device communicates with the Interactive House server. This is intended for devices such as simulations or external programs that do not use the BLE/USB bridge.

1. Overview

A REST device talks directly to the server using HTTP.

Base URL prefix:

/api/v1
Direction Endpoint Purpose
Device -> Server POST /api/v1/device-gateway/connect Register or update the device
Device -> Server POST /api/v1/device-gateway/{device_uuid}/heartbeat Tell the server the device is still alive
Device -> Server GET /api/v1/device-gateway/{device_uuid}/commands/next Fetch the next pending command
Device -> Server POST /api/v1/device-gateway/{device_uuid}/command-ack Report that a command was handled
Unit -> Server POST /api/v1/devices/{device_uuid}/commands Send a command to the device through the server

2. Required Flow

A REST device should follow this order:
Step 1: Connect to the server
Step 2: Start sending heartbeat regularly
Step 3: Poll commands/next regularly
Step 4: When a command is received, apply it locally
Step 5: Send command-ack with the actual resulting state

3. Connect

Endpoint

POST /api/v1/device-gateway/connect

Purpose

Register one or more devices with the server.

Minimum recommendation for a REST device

{
  "devices": {
    "RVC001": {
      "device_uuid": "RVC001",
      "type": "robot_vacuum",
      "transport": {
        "mode": "rest",
        "protocol": "rest"
      },
      "capabilities": {
        "cleaning": {
          "type": "boolean",
          "writable": true
        },
        "return_to_base": {
          "type": "boolean",
          "writable": true
        }
      },
      "state": {
        "cleaning": false,
        "return_to_base": false
      },
      "status": {
        "connected": true
      }
    }
  }
}

Typical success response

{
  "message": "Devices connected",
  "devices": { ... },
  "generated_uuids": {}
}

4. Heartbeat

Endpoint

POST /api/v1/device-gateway/{device_uuid}/heartbeat

Purpose

Tell the server that the device is still alive.

Request body

No JSON body is required in the current implementation.

Recommended interval

Every 5 seconds

Typical success response

{
  "device_uuid": "RVC001",
  "last_seen": "2026-03-30T10:47:07.150380+00:00",
  "state": {
    "cleaning": true,
    "return_to_base": false
  }
}

5. Poll Next Command

Endpoint

GET /api/v1/device-gateway/{device_uuid}/commands/next

Purpose

Ask the server if there is a pending command for this device.

Recommended interval

Every 0.5-1.0 seconds

When a command exists

{
  "device_uuid": "RVC001",
  "command": {
    "type": "COMMAND",
    "device_uuid": "RVC001",
    "state": {
      "cleaning": true
    }
  }
}

When no command exists

{
  "device_uuid": "RVC001",
  "command": null
}

6. Command Acknowledgement

Endpoint

POST /api/v1/device-gateway/{device_uuid}/command-ack

Purpose

Report that the command was processed and tell the server the actual resulting state.

Request body

{
  "status": "ok",
  "reported_state": {
    "cleaning": true,
    "return_to_base": false
  }
}

Important

reported_state should contain the actual state after the command was applied locally. The server uses this to update the stored device state.

Typical success response

{
  "message": "Command acknowledgement received",
  "device_uuid": "RVC001",
  "status": "ok",
  "reported_state": {
    "cleaning": true,
    "return_to_base": false
  }
}

7. Unit-side Command Flow

A Unit does not call the device-gateway routes.

The Unit still uses the normal server endpoint:

POST /api/v1/devices/{device_uuid}/commands

Example

{
  "state": {
    "cleaning": true
  }
}
The server decides how to deliver the command. For REST devices, the command is queued until the device fetches it through commands/next.

8. Important Rules

  • The device must send the same device_uuid consistently.
  • The device should connect first before sending heartbeat or polling commands.
  • The device should keep polling commands/next regularly while running.
  • The device should send command-ack after applying a command.
  • The device should treat command: null as “nothing to do right now”.
