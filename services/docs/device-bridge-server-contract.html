Device JSON Communication Format

This document describes the JSON formats used between devices, the bridge and the server. Devices send information about themselves and their capabilities. The bridge should not hardcode device types or capabilities. It should simply forward and process the data.

1. CONNECT message (Device -> Bridge)

This message is sent when a device or controller connects and wants to register or update one or more devices.

If the device has been registered before, it can usually send its device_uuid together with any changed fields instead of resending all static data every time.

Required fields

  • type must be "CONNECT"
  • devices contains all devices sent by the controller/device
{
  "type": "CONNECT",
  "devices": {
    "arduino-1": {
      "device_uuid": "arduino-1",
      "type": "controller",
      "transport": {
        "mode": "direct",
        "protocol": "ble",
        "address": "AA:BB:CC:DD:EE",
        "service": "HMSoft"
      },
      "status": {
        "connected": true
      },
      "last_seen": "2026-03-13T20:15:00Z"
    },
    "light-1": {
      "device_uuid": "light-1",
      "type": "light",
      "transport": {
        "mode": "via_controller",
        "controller_uuid": "arduino-1",
        "port": 13
      },
      "capabilities": {
        "power": {
          "type": "boolean",
          "writable": true
        }
      },
      "state": {
        "power": true
      },
      "last_seen": "2026-03-13T20:15:00Z"
    },
    "lamp-2": {
      "device_uuid": "lamp-2",
      "type": "light",
      "transport": {
        "mode": "direct",
        "protocol": "wifi",
        "endpoint": "192.168.1.45"
      },
      "capabilities": {
        "power": {
          "type": "boolean",
          "writable": true
        },
        "brightness": {
          "type": "integer",
          "min": 0,
          "max": 100,
          "writable": true
        }
      },
      "state": {
        "power": false,
        "brightness": 0
      },
      "last_seen": "2026-03-13T20:15:00Z"
    }
  }
}

Example of a smaller CONNECT payload for an already registered device

{
  "type": "CONNECT",
  "devices": {
    "arduino-1": {
      "device_uuid": "arduino-1",
      "status": {
        "connected": true
      },
      "last_seen": "2026-03-13T20:20:00Z"
    },
    "light-1": {
      "device_uuid": "light-1",
      "state": {
        "power": false
      },
      "last_seen": "2026-03-13T20:20:00Z"
    }
  }
}

2. Device Structure

Each entry inside devices represents one device.

  • device_uuid - unique identifier
  • type - device type (controller, light, fan, etc)
  • transport - how the device is reached
  • capabilities - object of supported features keyed by capability name
  • state - current state of the device
  • status - connection status
  • last_seen - timestamp of last communication

3. HEARTBEAT message (Device -> Bridge)

Heartbeat messages inform the server that a device is still alive.

{
  "type": "HEARTBEAT",
  "device_uuid": "arduino-1"
}

4. CONNECT_ACK message (Bridge -> Device)

Sent back to the device after a successful CONNECT request.

{
  "type": "CONNECT_ACK",
  "message": "Devices connected",
  "devices": {},
  "generated_uuids": {},
  "server_time": "2026-03-13T20:15:00Z"
}

5. COMMAND message (Server/Bridge -> Device)

This message is sent to a device when the server wants it to update its state.

Required fields

  • type must be "COMMAND"
  • device_uuid identifies which device should execute the command
  • state contains the state the device should change to
{
  "type": "COMMAND",
  "device_uuid": "light-1",
  "state": {
    "power": true
  }
}

Another example

{
  "type": "COMMAND",
  "device_uuid": "fan-1",
  "state": {
    "speed": 3
  }
}

6. COMMAND_ACK message (Device -> Bridge)

Sent back by the device after it has processed a command.

Required fields

  • type must be "COMMAND_ACK"
  • device_uuid identifies which device processed the command
  • status shows whether execution succeeded, failed or was partial
  • state shows the actual state after execution
{
  "type": "COMMAND_ACK",
  "device_uuid": "light-1",
  "status": "ok",
  "state": {
    "power": true
  }
}

Another example

{
  "type": "COMMAND_ACK",
  "device_uuid": "fan-1",
  "status": "ok",
  "state": {
    "speed": 3
  }
}

7. Transport Examples

BLE device

{
  "mode": "direct",
  "protocol": "ble",
  "address": "AA:BB:CC:DD:EE",
  "service": "HMSoft"
}

USB device

{
  "mode": "direct",
  "protocol": "usb_serial",
  "port": "COM5",
  "baud_rate": 9600
}

WiFi device

{
  "mode": "direct",
  "protocol": "wifi",
  "endpoint": "192.168.1.45"
}

Device controlled by controller

{
  "mode": "via_controller",
  "controller_uuid": "arduino-1",
  "port": 13
}

8. Capability Examples

Boolean capability

{
  "power": {
    "type": "boolean",
    "writable": true
  }
}

Integer capability

{
  "brightness": {
    "type": "integer",
    "min": 0,
    "max": 100,
    "writable": true
  }
}

Event capability

{
  "doorbell": {
    "type": "event",
    "writable": false
  }
}

9. Status Example

Online

{
  "connected": true
}

Offline

{
  "connected": false
}

10. Multiple Controllers

The system supports multiple controllers because the devices object can contain many entries.

{
  "type": "CONNECT",
  "devices": {
    "arduino-1": {
      "type": "controller",
      "device_uuid": "arduino-1"
    },
    "arduino-2": {
      "type": "controller",
      "device_uuid": "arduino-2"
    }
  }
}
Important:
The bridge should never hardcode device types or capabilities. Devices must send all required information themselves.
