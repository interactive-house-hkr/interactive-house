Device JSON Communication Format

This document describes the JSON formats used between devices, the bridge and the server. Devices send information about themselves and their capabilities. The bridge should not hardcode device types or capabilities; it simply forwards and processes the data.

1. CONNECT message (Device -> Bridge)

This message is sent when a device connects and wants to register or update a device.

This message is sent once per device. If an Arduino controls multiple devices, it should send multiple small CONNECT messages instead of one large payload. 

{
  "type": "CONNECT",
  "devices": {
    "light-1": {
      "device_uuid": "light-1",
      "type": "light",
      "capabilities": {
        "power": {
          "type": "boolean",
          "writable": true
        }
      },
      "state": {
        "power": true
      }
    }
  }
}

2. Device Structure

Each entry inside devices represents one device.

  • device_uuid - unique identifier
  • type - device type (light, fan, door, etc)
  • capabilities - object of supported features keyed by capability name
  • state - current state of the device
  • status - connection status
  • last_seen - timestamp of last communication

3. HEARTBEAT message (Device -> Bridge)

Heartbeat messages inform the server that a device is still alive.

{
  "type": "HEARTBEAT",
  "device_uuid": "arduino-1"
}

4. COMMAND message (Server/Bridge -> Device)

This message is sent to a device when the server wants it to update its state.

Required fields

  • type must be "COMMAND"
  • device_uuid identifies which device should execute the command
  • state contains the state the device should change to
{
  "type": "COMMAND",
  "device_uuid": "light-1",
  "state": {
    "power": true
  }
}

Another example

{
  "type": "COMMAND",
  "device_uuid": "fan-1",
  "state": {
    "speed": 3
  }
}

5. COMMAND_ACK message (Device -> Bridge)

Sent back by the device after it has processed a command.

Required fields

  • type must be "COMMAND_ACK"
  • device_uuid identifies which device processed the command
  • status shows whether execution succeeded, failed or was partial
  • reported_state shows the actual state after execution
{
  "type": "COMMAND_ACK",
  "device_uuid": "light-1",
  "status": "ok",
  "reported_state": {
    "power": true
  }
}

Another example

{
  "type": "COMMAND_ACK",
  "device_uuid": "fan-1",
  "status": "ok",
  "reported_state": {
    "speed": 3
  }
}

6. Capability Examples

Boolean capability

{
  "power": {
    "type": "boolean",
    "writable": true
  }
}

Integer capability

{
  "brightness": {
    "type": "integer",
    "min": 0,
    "max": 100,
    "writable": true
  }
}

Event capability

{
  "doorbell": {
    "type": "event",
    "writable": false
  }
}

7. Status Example

Online

{
  "connected": true
}

Offline

{
  "connected": false
}
Important:
The bridge should never hardcode device types or capabilities. Devices must send all required information themselves.
